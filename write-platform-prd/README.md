# write-platform-prd

用于把**平台型 / 中后台 / 多角色 / 规则驱动 / 强流程协同**的需求，整理成一份：

- 可评审
- 可研发
- 可测试
- 可上线演进

的 PRD。

它不是单纯润色文档，也不是把原始材料重新排版。它的核心价值是：**自动补齐平台型 PRD 最容易漏掉的实现层、协作层和治理层内容。**

## 适用场景
- 多角色协同平台
- 管理后台 / 运营后台 / 商家后台
- 审核流、接单流、履约流、结算流
- 规则配置平台
- 前台 + 后台协同系统
- 状态机复杂的业务
- 有跨模块联动的数据流程
- 需要考虑灰度、兼容、存量数据的需求

## 不适用场景
- 纯 C 端活动页
- 轻量营销页面
- 纯文案润色
- 商业方案 / 路演稿
- 用户访谈纪要原文整理
- 纯技术架构设计文档
- 专门做研发逻辑挑刺的审查类场景

## 文件说明
### `SKILL.md`
核心 skill 文件，定义平台型 PRD 的输出目标、需求分级、章节结构和重点补充规则。

### `how-to-use.md`
给使用者看的说明文件，解释这个 skill 适合什么时候用、可以直接喂什么材料、L1/L2/L3 怎么理解、材料不完整时会怎么处理。

### `output-template.md`
统一输出骨架，按 L1 / L2 / L3 三档提供标准结构，方便团队统一产出样式。

### `example-output.md`
示例输出，展示一份高质量平台型 PRD 最终应该长什么样。

## 需求分级说明
### L1：轻量迭代
适合文案调整、字段新增/删除、筛选项新增、小交互优化、小范围规则微调。

### L2：标准功能
适合新增一个页面、模块内新增功能、有简单流程变化、有基础权限变化、有基础后台支持。

### L3：平台型重需求
适合新模块、新流程、强状态机、跨模块联动、多角色协同、规则配置、涉及灰度/兼容/存量数据，以及资金、审核、履约、结算等核心链路。

一个重要原则：**不是看字数，而是看复杂度和风险。** 只要命中跨模块联动、状态机变化、核心对象变化、权限模型变化、规则配置变化、上下游依赖、存量兼容、资金/审核/履约/结算等核心链路，就默认按 L3 处理。

## 风险提示
这个 skill 会在信息不完整时继续成文，但会把关键不确定点标记为：
- `[待确认]`
- `[AI 推测 - 待确认]`
- `[实现口径待明确]`
- `[财务口径待明确]`
- `[法务口径待明确]`

这些标记不能忽略，尤其在资金、法务、PII、权限边界、状态机逆向流转、灰度与兼容、接口升级等场景中必须二次确认。

## 一句话总结
`write-platform-prd` 不是帮你“写一份更长的需求文档”，而是帮你把**平台型需求里最容易漏掉的实现层、治理层和协作层问题先显性化**。
