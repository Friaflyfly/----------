# write-platform-prd 使用说明

这个文件不是给 PD 的写作模板。它是给“使用这个 skill 的人”看的，目的是帮助大家更稳定地使用 `write-platform-prd`。

## 这个 skill 是干什么的
它用于把**平台型 / 中后台 / 多角色 / 规则驱动 / 强流程协同**的需求，整理成一份：
- 可评审
- 可研发
- 可测试
- 可上线演进

的 PRD。

它的价值在于：**自动补齐平台型 PRD 最容易漏掉的那一大半东西**，例如角色与权限、业务对象与对象关系、状态机、规则与配置生效机制、后台操作与兜底、异常场景、上下游依赖、灰度与兼容、PII / 脱敏 / 明文审计、验收标准、成功指标等。

## 可以直接拿什么材料来写
不要求输入符合固定模板。以下材料都可以直接拿来写：
- 原始 PRD 初稿
- 飞书文档片段
- 会议纪要
- 原型说明
- 页面清单
- 功能点列表
- 业务规则草稿
- 口头整理后的需求文本

也就是说：**材料可以不完整，也可以不工整。** 这个 skill 的任务就是先从混乱输入里抽结构，再整理成平台型 PRD。

## 输入不完整时会怎么处理
当材料不完整时，这个 skill 不会停住不动，而会按下面的方式继续推进：
1. 能明确写的部分直接写
2. 高概率属于平台型通用要求的部分，会补出结构
3. 无法确认但明显关键的内容，会标记为：
   - `[待确认]`
   - `[AI 推测 - 待确认]`
   - `[实现口径待明确]`
   - `[财务口径待明确]`
   - `[法务口径待明确]`

最终文档末尾会输出：
## 【待确认清单】

## 怎么理解 L1 / L2 / L3
### L1：轻量迭代
适合文案调整、字段新增/删除、筛选项新增、小交互优化、小范围规则微调。

### L2：标准功能
适合新增一个页面、某模块内新增功能、有简单流程变化、有权限变化、有基础后台支持。

### L3：平台型重需求
适合新模块、新流程、强状态机、跨模块联动、多角色协同、规则配置、涉及灰度/兼容/存量数据，以及资金、审核、履约、结算等核心链路。

一个很重要的原则：即使用户只写了几句话，只要命中跨模块联动、状态机变化、核心业务对象变化、权限模型变化、规则配置变化、上下游系统依赖、存量数据兼容、资金 / 审核 / 履约 / 结算等核心链路，也会按 **L3** 处理。

## 推荐怎么用
### 方式一：直接扔原始材料
最常见也最实用。直接把会议纪要、原型说明、草稿需求丢进去，让它整理成 PRD。

### 方式二：补一句重点关注方向
例如：
- 这次重点看状态机和审核权限
- 这次重点看后台配置与生效机制
- 这次重点看结算规则与数据口径
- 这次重点看 PII 脱敏和查看明文权限
- 这次重点看灰度和老数据兼容

### 方式三：先出完整稿，再自己删减
对于复杂需求，建议先让它按平台型完整结构出一版，然后再根据实际评审对象删减。

## 用这个 skill 时要特别留意什么
1. 不要把 AI 推测当既定事实。文档里凡是出现 `[AI 推测 - 待确认]`、`[待确认]`、`[实现口径待明确]`，都应该在评审时重点确认。
2. 财务 / 法务内容不能让 AI 自己定。比如结算比例、手续费率、赔偿金额、税费规则等。
3. 平台型需求要重点检查逆向逻辑。不仅看怎么创建、提交、通过，还要看能不能撤回、谁能撤销、是否需要冲抵 / 冲正、失败后怎么回滚。
4. PII / 敏感信息不要只写一句“脱敏”。必须重点确认谁能看脱敏、谁能看明文、明文查看是否留痕、导出 / 接口 / 列表页口径是否一致。

## 一句话原则
**这个 skill 不是替你“写漂亮文档”，而是替你把平台型需求里最容易漏掉的实现层和协作层问题先显性化。**
