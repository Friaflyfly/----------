# 钱包与账号中心改造方案

> 本文档面向 AI 编码助手，用于指导对"我的钱包"与"账号中心"两个模块的信息架构与导航重构。AI 在实施前请先阅读当前代码库，理解现有路由、组件、状态管理方案，再按本方案落地。

---

## 一、背景与目标

### 现状问题
当前"我的钱包"采用 Tab 形式组织内部子功能（余额、明细、优惠券等），存在以下问题：

1. Tab 适合"同对象的不同视图"，不适合"功能并列、路径独立"的场景
2. 所有子功能视觉权重一样，高频与低频功能挤在一起
3. 子功能无独立路由，无法深链、无法外部跳转
4. 扩展性差，新增功能需要继续挤 Tab
5. 与"账号中心"形态不一致，用户心智不统一

### 改造目标
1. 将"我的钱包"提升为独立模块，与"账号中心"同级
2. 钱包内部采用左侧菜单 + 内容区结构，每个子功能独立路由
3. 重新设计钱包首页为概览型 Dashboard，而非空菜单或明细列表
4. 明确钱包与账号中心的功能边界
5. 保证老链接平滑迁移，不影响线上用户

---

## 二、改动范围

### 涉及模块
- 全局导航（右上角用户下拉菜单）
- 我的钱包（Wallet）模块——整体重构
- 账号中心（Account）模块——边界调整、菜单梳理
- 路由配置
- 埋点配置

### 不涉及
- 业务主流程（下单、搜索、列表等）
- 钱包底层资金接口（仅前端结构调整）
- 账号鉴权与登录逻辑

---

## 三、全局导航变更

### 用户下拉菜单结构（右上角头像）

```
┌─────────────────────────────┐
│ [头像]                       │
│ 用户昵称                     │
│ VIP 等级 / 角色标签           │
├─────────────────────────────┤
│ 👤 账号中心                  │
│ 💰 我的钱包      ¥1,234.56  │  ← 余额直接展示
│ 📋 我的订单                  │
│ 🔔 消息中心                  │
├─────────────────────────────┤
│ ⚙️ 偏好设置                  │
│ ❓ 帮助与反馈                 │
├─────────────────────────────┤
│ 🚪 退出登录                  │
└─────────────────────────────┘
```

### 要点
- 钱包余额在下拉菜单中直接展示，作为信息+入口的复合元素
- 余额为 0 或未开通时显示"未激活"或不显示金额
- 主导航（左侧或顶部）只放业务功能，"账号中心"和"我的钱包"不进主导航
- 移动端：用户下拉变为抽屉或个人中心 Tab 页

## 四、路由设计

### 钱包模块路由

| 路径 | 页面 | 说明 |
|------|------|------|
| `/wallet` | 钱包概览 | 默认落地页，Dashboard 形态 |
| `/wallet/balance` | 账户余额详情 | 余额历史、冻结金额、待入账 |
| `/wallet/recharge` | 充值 | 充值入口与历史 |
| `/wallet/withdraw` | 提现 | 提现入口与历史 |
| `/wallet/transactions` | 交易明细 | 流水视角，按时间排序 |
| `/wallet/bills` | 账单 | 周期视角，按月汇总 |
| `/wallet/invoices` | 发票管理 | 开票、抬头、申请记录 |
| `/wallet/coupons` | 优惠券 | 可用、已用、已过期 |
| `/wallet/gift-cards` | 礼品卡 | 如有则保留 |
| `/wallet/points` | 积分 | 如有则保留 |
| `/wallet/payment-methods` | 支付方式 | 绑定的银行卡、第三方 |
| `/wallet/security` | 支付安全 | 支付密码、限额、风控设置 |

### 账号中心路由

| 路径 | 页面 | 说明 |
|------|------|------|
| `/account` | 账号概览 | 或默认跳转 `/account/profile` |
| `/account/profile` | 基本资料 | 头像、昵称、生日等 |
| `/account/verification` | 实名认证 | 身份信息绑定 |
| `/account/security` | 账号安全 | 登录密码、二次验证、设备管理 |
| `/account/contacts` | 手机/邮箱 | 联系方式绑定与变更 |
| `/account/oauth` | 第三方账号 | 微信、Google 等绑定 |
| `/account/notifications` | 消息通知 | 推送/邮件偏好 |
| `/account/privacy` | 隐私设置 | 数据权限、可见性 |
| `/account/danger` | 账号注销 | 高危操作单独成页 |

### 路由迁移（重要）

原 Tab 形态的 URL 必须做 301/302 重定向：

```
原 /wallet?tab=balance      → /wallet/balance
原 /wallet?tab=transactions → /wallet/transactions
原 /wallet?tab=coupons      → /wallet/coupons
... 其余以此类推
```

AI 实施时请检索代码中所有 `?tab=` 形式的钱包链接，统一替换。

### 5.2 钱包概览页（/wallet）—— 重点设计

落地页不是菜单列表，而是 Dashboard：

**区块 1：余额卡片（顶部突出）**
- 大字号显示总余额
- 副信息：可用余额、冻结金额、待入账
- 右侧两个主操作按钮：充值、提现
- 视觉权重最高

**区块 2：本期数据（4 个小卡片横排）**
- 本月收入
- 本月支出
- 本月节省（优惠券抵扣 + 积分抵扣）
- 待入账金额

**区块 3：最近交易（列表，5-10 条）**
- 显示时间、类型、金额、状态
- 右上角"查看全部"跳转 `/wallet/transactions`

**区块 4：我的凭证（横向卡片）**
- 可用优惠券数量
- 积分余额
- 即将过期提醒（红点 + 数字）

**区块 5：快捷入口（次要）**
- 开发票、查账单、绑定支付方式

### 5.3 左侧菜单组件规范

- 选中态高亮，未选中态浅色
- 分组之间有分隔线或分组标题
- 支持折叠（移动端必须）
- 当前页 active 状态与路由同步
- 余额、优惠券数量等可在菜单项后显示徽标

## 六、钱包与账号中心边界

### 划分原则
> **账号中心管"我是谁"，钱包管"我有什么"。**

### 容易混淆的项目处理

| 功能 | 归属 | 理由 |
|------|------|------|
| 登录密码 | 账号中心 | 身份验证 |
| 支付密码 | 钱包 | 资金安全 |
| 手机号绑定 | 账号中心 | 身份属性 |
| 实名认证 | 账号中心 | 身份属性，但钱包内涉及支付时引导跳转 |
| 银行卡管理 | 钱包 | 资金工具 |
| 第三方账号绑定 | 账号中心 | 登录方式 |
| 发票抬头 | 钱包（发票管理内） | 资金凭证 |
| 收货地址 | 账号中心或独立模块 | 不放钱包 |
| 二次验证（短信/TOTP） | 账号中心 | 通用账号安全 |
| 支付限额 | 钱包 | 资金风控 |

---

## 七、组件复用建议

AI 实施时请优先复用现有组件，必要时新建：

1. **左侧菜单组件**：账号中心如有现成菜单组件，抽离为通用 `<SideMenu>`，钱包复用
2. **面包屑组件**：通用，全局共用
3. **数据卡片组件**：余额卡、统计卡可抽象为 `<MetricCard>`
4. **交易列表组件**：明细、账单、充值/提现记录可能共用列表 + 筛选器
5. **空状态组件**：无优惠券、无明细等场景统一

---

## 八、状态管理与数据

1. 钱包数据建议独立 store/slice（如 `walletStore`），不要塞进 user store
2. 余额、凭证数量等需要在全局导航显示的数据，做轻量缓存 + 进入钱包模块时刷新
3. 跨页面共享的数据（如优惠券列表）做 store 级别缓存，避免每个子页面重复请求
4. 接口失败要有明确的错误态，不要静默失败（涉及资金，用户敏感）

---

## 九、返回与跳转体验

独立模块后必须处理好"返回原业务"的体验：

1. 钱包/账号中心页面顶部需有清晰的返回按钮或面包屑
2. 从业务页面跳转钱包时，记录来源（URL 参数或 referrer），返回时回到原页面
3. 支付完成 → 查看明细 → 返回业务，全链路要顺畅
4. 浏览器后退按钮行为正常（每个子页是独立历史记录）

---

## 十、移动端适配

1. PC 端的左侧菜单，在移动端变为：
   - 顶部 Tab 横滑（如果项目少）
   - 或抽屉式菜单（汉堡按钮触发）
   - 或独立的二级页面 + 列表入口（推荐，符合移动端心智）

2. 钱包概览页在移动端按区块纵向排列，余额卡片占满宽度
3. 充值、提现等关键操作按钮要足够大，符合点击区域规范
4. 表格类页面（明细、账单）在移动端转为卡片列表

---

## 十一、埋点

1. 原 Tab 切换事件改为页面访问事件（PV），事件名重新规划
2. 关键转化路径埋点：
   - 进入钱包 → 进入充值 → 完成充值
   - 进入钱包 → 进入优惠券 → 使用优惠券（跳出钱包到业务）
3. 提前与数据团队对齐口径，做好新老事件映射表

---

## 十二、验收标准

实施完成后需满足以下条件：

### 功能验收
- [ ] 用户下拉菜单中"账号中心"与"我的钱包"为同级独立入口
- [ ] 用户下拉菜单中钱包项显示当前余额
- [ ] 钱包默认落地页为概览 Dashboard，不是菜单列表
- [ ] 钱包左侧菜单按"概览/凭证/记录/设置"分组
- [ ] 每个钱包子功能有独立 URL，可直接访问、可分享
- [ ] 原 Tab 形式的 URL 全部 301 重定向到新路径
- [ ] 账号中心结构同步梳理，与钱包边界清晰
- [ ] 移动端有独立适配方案，不是直接缩放 PC 版

### 技术验收
- [ ] 路由配置集中管理，无散落硬编码
- [ ] 左侧菜单组件抽象复用，账号中心和钱包共用
- [ ] 钱包数据有独立 store，不混入 user store
- [ ] 旧链接重定向通过自动化测试覆盖
- [ ] 浏览器前进后退行为正常
- [ ] 加载态、错误态、空态完整

### 体验验收
- [ ] 从业务页面跳转钱包后能正确返回来源
- [ ] 主要操作（充值、提现、查明细）3 步以内可达
- [ ] 钱包首页 3 秒内能看到核心信息（余额、本月支出、可用券）
- [ ] 切换页面有过渡动画，不闪烁

## 十三、实施顺序建议

考虑到改动较大，建议分阶段上线：

**阶段 1：导航层调整（影响最广，工作量适中）**
- 用户下拉菜单重构
- 钱包入口提升为独立路由
- 保留原 Tab 内部结构，但加 URL 重定向

**阶段 2：钱包概览页改造（用户感知最强）**
- 实现 Dashboard 形态的 /wallet 首页
- 余额、本月数据、最近交易模块

**阶段 3：内部菜单与子页面拆分（工程量最大）**
- 左侧菜单组件
- 每个子功能独立成页
- 子页面之间的导航与返回

**阶段 4：账号中心同步梳理 + 边界调整**
- 把误放在账号中心的资金相关功能挪到钱包
- 反之亦然

**阶段 5：移动端适配 + 埋点更新**

每个阶段都可以独立上线，互不阻塞。

---

## 十四、给 AI 实施者的提醒

1. **先读代码再动手**：本方案是产品设计层面，具体落地需要结合现有技术栈、组件库、路由方案。请先 grep 出现有钱包、账号中心的所有相关文件，理解现状再开始。

2. **不要破坏现有 API 契约**：本次是前端结构重构，不应修改后端接口。如发现接口不支持新结构（例如缺少"本月统计"接口），请列出需求而非自行修改后端。

3. **小步快跑**：分阶段提交 PR，每个阶段独立可测、可回滚。不要一个 PR 改完所有东西。

4. **关注移动端**：每改一个 PC 页面，同步确认移动端形态，不要留尾巴。

5. **遇到边界模糊的情况询问**：例如某个功能应该放钱包还是账号中心、是否要保留旧 Tab 入口做兼容期等，列出来问，不要自行决定。
