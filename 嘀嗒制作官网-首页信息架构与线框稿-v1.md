# 嘀嗒制作官网-首页信息架构与线框稿 V1

## 1. 目标

首页的任务不是把所有规则一次讲完，而是完成 4 件事：

1. 让用户快速理解「嘀嗒制作是什么」
2. 让用户知道「适不适合我」
3. 让用户快速看到「有哪些标准产品包、多少钱、多久交付」
4. 让用户在下单前对流程和保障建立基本信任

因此建议采用：

- 首页：讲清价值、产品概览、流程概览、保障概览、FAQ 概览
- 详情页：承接产品包细节、价格档位、规则细则、下单动作

### 02. 嘀嗒制作是什么

**目标**

- 解释产品定位
- 说清解决什么问题
- 说清面向哪些客户

**建议标题**

- 嘀嗒制作是什么

**建议文案骨架**

- 产品定位：一个提供标准化图片与视频制作服务的在线下单入口
- 解决问题：帮助客户减少反复沟通、缩短制作决策时间、降低交付不确定性
- 面向客户：电商商家、品牌营销团队、本地生活商家、内容运营团队、中小企业

**建议展示方式**

三列价值卡：

1. 需求更好提
2. 制作更省心
3. 交付更可控

**文案示例**

- 需求更好提：通过标准产品包与需求模板，减少“不会提需求”的沟通成本
- 制作更省心：下单后按固定流程推进，节点清晰、责任明确
- 交付更可控：交付内容、修改次数、验收标准提前约定，避免扯皮

### 05. 下单与制作流程

**目标**

- 降低用户对“买完之后会发生什么”的不确定感

**建议标题**

- 下单到交付，流程一目了然

**建议采用 6 步流程**

1. 选择产品包
2. 提交需求与素材
3. 平台开始制作
4. 阶段审核与反馈
5. 验收并交付
6. 售后支持

**每一步一句话解释**

- 选择产品包：根据场景选择图片包或视频包，确认服务规格与价格
- 提交需求与素材：按模板填写需求，并上传参考资料或原始素材
- 平台开始制作：制作团队按下单内容进入排期与制作
- 阶段审核与反馈：按约定节点提交版本，用户可提出修改意见
- 验收并交付：验收通过后提供最终交付文件
- 售后支持：在约定范围内提供修改、下载与售后协助

**线框**

```text
[1 选择产品包] -> [2 提交需求] -> [3 开始制作] -> [4 审核反馈]
                                -> [5 验收交付] -> [6 售后支持]
```

### 06. 交付保障

**目标**

- 回答“靠不靠谱”
- 解决用户对交付边界最敏感的问题

**建议标题**

- 交付规则提前说清，合作更省心

**建议分成 4 张保障卡**

#### 保障 1：修改机制

- 每个产品包包含明确修改次数
- 合理范围内修改按产品规则执行
- 超出原始需求范围的新增内容，按增项或二次报价处理

#### 保障 2：审核规则

- 按下单需求、参考风格、规格要求进行审核
- 审核以产品包约定内容为准，不以临时新增口径为准

#### 保障 3：交付标准

- 明确交付格式、尺寸/时长、清晰度、命名与打包方式
- 最终交付物以产品详情页约定为准

#### 保障 4：版权与素材要求

- 用户需确保提交素材具有合法使用权
- 平台仅基于客户提交资料与约定需求完成制作
- 成品版权/使用权范围按订单约定执行

### 07. 常见问题 FAQ

**目标**

- 处理临门一脚问题
- 减少用户离开页面去咨询

**建议标题**

- 常见问题

**建议首批 FAQ**

1. 制作周期多久？
2. 支持开发票吗？
3. 不满意可以退款吗？
4. 怎么验收交付结果？
5. 交付后在哪里下载？
6. 可以修改几次？
7. 素材没准备齐可以下单吗？
8. 非标准需求怎么办？

**FAQ 文案骨架**

- 制作周期多久：不同产品包周期不同，以下单页展示的预计交付周期为准
- 支持开发票吗：支持，开票规则与抬头信息可在下单后确认
- 不满意可以退款吗：标准产品包退款规则按制作进度与服务条款执行
- 怎么验收：按下单需求、交付规格和约定标准进行验收
- 交付后在哪里下载：验收完成后可在订单页或交付页下载最终文件
- 可以修改几次：按所购买产品包说明执行
- 素材没准备齐可以下单吗：可先查看素材要求，部分产品包需素材齐全后才能进入制作
- 非标准需求怎么办：可联系顾问或提交定制需求

## 4. 首页内容层级建议

建议按照下面顺序排布：

1. Hero 首屏
2. 嘀嗒制作是什么
3. 产品包总览
4. 热门产品包
5. 下单与制作流程

7. FAQ
8. 底部 CTA

这个顺序对应用户决策路径：

- 先理解是什么
- 再判断适不适合自己
- 再看卖什么
- 再看怎么买
- 再看靠不靠谱
- 最后决定是否下单
