# write-prototype-brief

用于把**平台型 / 中后台 / 多角色 / 强流程页面**的需求，整理成一份适合 AI coding 工具直接消费的 **Prototype Brief（原型说明稿）**。

它不是完整 PRD，也不是单纯的页面文案说明。  
它的核心价值是：**在正式写 PRD 之前，先把页面级信息架构、字段、交互、状态、异常、权限和实现提示收口，便于快速生成 HTML 原型并开展评审。**

---

## 适用场景

这个 skill 特别适合以下类型的页面或模块：

- 管理后台 / 运营后台 / 商家后台页面
- 列表页
- 详情页
- 新建 / 编辑表单页
- 审核页
- 配置页
- 工作台 / 看板页
- 协同页
- 账本 / 明细页
- 需要先用 Cursor、Claude、Copilot 等 AI coding 工具生成 HTML 原型的页面

典型例子：

- 项目大厅页
- 订单审核页
- 收益账本页
- 规则配置页
- 学员入驻审核页
- 商户详情页
- 批量任务管理页

---

## 不适用场景

以下内容不建议使用这个 skill：

- 完整平台型 PRD
- 需要深度描述跨模块业务规则的文档
- 研发逻辑审查 / 挑 bug
- 技术架构设计文档
- 纯视觉规范 / 品牌规范
- 纯 C 端活动页创意文案

如果目标是输出正式需求文档，应使用 `write-platform-prd`。  
如果目标是从研发视角审查逻辑漏洞，应使用 review 类 skill。

---

## 这个 skill 解决什么问题

很多页面需求的问题不是“没有页面”，而是：

- 页面目标不清楚
- 谁用、谁能看、谁能点不清楚
- 状态与按钮关系没写
- 列表字段和详情字段对不上
- 批量操作、异步任务、搜索重置这些边界没写
- AI coding 工具只能猜，最后生成的 HTML 返工很多

`write-prototype-brief` 的目标就是先把这些页面级问题收口，输出一份：

- 可直接喂给 AI coding 工具
- 可给 UI / 前端继续深化
- 可快速进入页面评审

的中间层文档。

## 它和 write-platform-prd 的关系

两者不是替代关系，而是前后衔接关系。

### `write-prototype-brief`
前置使用。  
适合在需求还在收敛、但已经想先看页面方案时使用。

它重点解决：
- 页面目标
- 页面骨架
- 字段规则
- 交互与状态
- 异常与权限
- 给 AI coding 工具的实现提示

### `write-platform-prd`
后置使用。  
适合在页面方案相对清楚后，补齐正式需求文档。

它重点解决：
- 业务对象
- 状态机
- 规则配置
- 上下游依赖
- 灰度兼容
- 数据口径
- 验收标准
- 生命周期与演进

一句话理解：

**Prototype Brief 先把页面做出来，Platform PRD 再把系统逻辑写完整。**

## 文件说明

### `SKILL.md`
核心 skill 文件。  
定义 Prototype Brief 的目标、边界、输出结构、字段规则、交互规则、异常态、A11y、给 AI coding 工具的实现提示等。

### `README.md`
入口说明。  
帮助团队快速理解这个 skill 是干什么的、适合什么场景、和 `write-platform-prd` 的关系。

### `how-to-use.md`
使用说明。  
重点说明：可以喂什么材料、页面类型怎么判断、材料不完整时如何处理、如何让 AI coding 工具输出更稳定。

### `output-template.md`
统一输出骨架。  
帮助不同人写出的 Prototype Brief 结构尽量一致。

### `example-output.md`
示例输出。  
展示一份高质量 Prototype Brief 应该长什么样。

### `prototype-output-standard.md`
原型输出标准。  
沉淀产品 → UI / 产品 → AI coding 的交付细则，约束字段完整性、交互完整性、标注与版本要求。

## 需求分级理解

Prototype Brief 不一定都一样重。  
建议按页面复杂度使用：

### L1：轻页面
适合：
- 简单列表页
- 小表单
- 文案调整
- 筛选项新增
- 小交互优化

特点：
- 输出短
- 更关注页面区块、字段、按钮和异常态

### L2：标准页面
适合：
- 有主流程的详情页
- 新建 / 编辑页
- 基础审核页
- 一般配置页

特点：
- 会写角色差异、字段规则、交互、状态、异常、实现提示

### L3：复杂平台页面
适合：
- 多角色页面
- 强状态机页面
- 配置发布页
- 批量任务页
- 审核 / 财务 / 权限类页面
- 有敏感字段、异步任务、跨页面联动的页面

特点：
- 输出完整 Prototype Brief
- 会重点补：
  - 面包屑与入口出口
  - 字段一致性
  - 语义化颜色
  - 批量操作计数
  - 异步任务进度反馈
  - 明文查看交互
  - 副作用说明
  - A11y / 语义化 HTML 提示

## 可以直接拿什么材料来写

不要求输入符合固定模板。  
以下材料都可以直接拿来生成 Prototype Brief：

- 会议纪要
- 页面草图
- 原型描述
- 飞书文档片段
- PRD 中的一小段页面说明
- 一段口头需求
- 一个页面名称 + 简短目标
- 已有页面截图 + 改版诉求

也就是说：

**材料可以不完整，但应该至少能判断出页面目标。**

## 输出时默认会重点补哪些内容

对于平台型页面，默认重点补充：

- 页面目标
- 使用角色
- 页面入口 / 出口
- 面包屑路径
- 页面区块结构
- 字段完整性
- 表单 placeholder / 示例值 / 错误提示
- 列表 / 详情字段一致性
- 枚举值与联动逻辑
- 状态语义化颜色
- 异常态 / 空态 / 权限态
- 批量操作计数与反馈
- 异步任务进度反馈
- 行级操作“更多”收纳
- 搜索重置默认态
- 附件字段规则
- 明文查看交互
- Design System 复用
- 给 AI coding 工具的实现提示
- A11y / 语义化 HTML
- 待确认项

## 风险提示

这个 skill 会在信息不完整时继续成文，但会把关键不确定点标记为：

- `[待确认]`
- `[AI 推测 - 待确认]`
- `[实现口径待明确]`

这些内容不能直接默认成事实，尤其在下面这些场景中要重点确认：

- 角色差异
- 状态流转
- 枚举值
- 敏感字段脱敏
- 批量任务反馈
- 搜索默认态
- 明文查看审计
- 页面副作用
- 响应式范围
- i18n 占位

## 最佳实践

### 最稳的用法
1. 先丢原始页面需求生成一版 Prototype Brief
2. 重点看 `待确认项`
3. 用 Brief 喂给 Cursor / Claude / Copilot 出 HTML
4. 看页面评审一轮
5. 页面方案收敛后，再补 `write-platform-prd`

### 最容易翻车的用法
- 直接一句“帮我做个后台页面”就让 AI coding 工具开画
- 不写角色
- 不写状态
- 不写异常态
- 不看待确认项
- 不约束响应式、语义化和组件复用

---

## 一句话总结

`write-prototype-brief` 不是帮你“写更长的页面说明”，而是帮你把**平台型页面最容易漏掉的字段、交互、状态、异常和实现提示先收口**，让 AI coding 工具更稳定地产出可评审的 HTML 原型。
