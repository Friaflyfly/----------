# prototype-output-standard.md

本文件用于统一 `write-prototype-brief` 的输出质量标准。  
它不是正式 PRD，也不是 UI 视觉规范手册，而是用于约束：

- 产品 → AI coding 工具
- 产品 → UI
- 产品 → 前端

之间的页面级交付质量。

目标是减少以下问题：

- 字段定义不完整
- 列表 / 详情 / 编辑字段不一致
- 按钮状态和交互逻辑缺失
- 异常态 / 权限态未覆盖
- 批量操作、异步任务、搜索重置逻辑不清
- 原型虽能看，但无法直接落地

---

# 1. 适用范围

适用于以下页面或模块：

- 列表页
- 详情页
- 新建 / 编辑页
- 审核页
- 配置页
- 工作台 / 看板页
- 协同页
- 账本 / 明细页
- 批量任务页

---

# 2. 交付标准总原则

一份合格的 Prototype Brief / 原型输出，必须满足：

1. 页面目标清晰  
2. 角色边界清晰  
3. 页面入口 / 出口清晰  
4. 信息架构清晰  
5. 字段规则完整  
6. 状态与交互闭环  
7. 异常态 / 空态 / 权限态完整  
8. 组件复用优先  
9. 标注与版本可追溯  
10. 能直接被 AI coding 工具、UI、前端消费

---

# 3. 页面基础信息标准

每份输出至少必须包含以下基础信息：

## 3.1 页面基础信息
- 页面名称
- 页面类型
- 所属模块
- 所属业务链路
- 页面目标
- 页面不解决的问题（如适用）

## 3.2 角色信息
- 使用角色
- 角色差异
- 不同角色可见区块
- 不同角色可见字段
- 不同角色可操作按钮

## 3.3 页面层级信息
- 页面入口
- 页面出口
- 返回路径
- 成功后跳转路径
- 面包屑路径
- 是否存在多入口
- 多入口时返回逻辑是否一致

---

# 4. 字段定义标准

## 4.1 所有字段必须明确
每个字段至少应尽量写清：

- 字段名称
- 字段含义
- 展示位置
- 字段类型
- 是否必填
- 默认值
- 是否可编辑
- 是否可复制
- 是否脱敏
- 是否联动
- 备注

## 4.2 表单字段补充要求
若字段为输入型字段，必须尽量补充：

- placeholder
- 示例值
- 错误提示文案
- 字符限制（最小 / 最大长度）
- 输入类型（文本 / 数字 / 邮箱 / 手机号 / 密码等）
- 是否允许空格
- 是否允许特殊字符
- 是否支持即时校验
- 是否支持失焦校验

## 4.3 附件 / 图片上传字段补充要求
若字段为附件 / 图片上传，必须尽量补充：

- 文件格式要求
- 单文件大小限制
- 总大小限制（如适用）
- 是否支持多选
- 是否支持拖拽上传
- 是否支持预览
- 是否支持下载
- 是否支持删除 / 替换
- 上传失败提示文案

## 4.4 敏感字段补充要求
若字段涉及手机号、身份证号、银行卡号、地址、金额等敏感内容，必须尽量补充：

- 列表页展示规则
- 详情页展示规则
- 是否脱敏
- 不同角色可见粒度
- 是否允许明文查看
- 明文查看交互方式
- 是否需要审计
- 是否允许复制 / 导出

---

# 5. 一致性标准

## 5.1 同一字段必须一致
同一字段在以下场景中应尽量保持一致：

- 新建页
- 编辑页
- 列表页
- 详情页
- 搜索区
- 筛选区
- 导出字段（如适用）

## 5.2 禁止出现的情况
- 列表页字段名称与详情页字段名称不一致
- 列表页有关键字段，详情页无对应展示
- 新建页有字段，编辑页缺失
- 同一枚举值在不同页面文案不一致
- 同一状态颜色不一致
- 同一字段格式不一致（如金额、时间、手机号）

---

# 6. 枚举值与联动逻辑标准

## 6.1 枚举字段必须明确
每个枚举字段至少应尽量写清：

- 全部枚举值
- 对应显示文案
- 默认值
- 排序顺序
- 是否影响按钮状态
- 是否影响字段显隐
- 是否影响页面区块显隐

## 6.2 语义化颜色
若状态 / 枚举通过视觉表达，必须尽量定义：

- 成功态颜色
- 失败态颜色
- 警告 / 待处理态颜色
- 禁用态颜色
- 同一状态在标签、文字、图标中的颜色是否一致

---

# 7. 页面交互标准

## 7.1 按钮交互
每个关键按钮至少应尽量写清：

- 按钮名称
- 按钮位置
- 触发动作
- 可点击条件
- 置灰条件
- 置灰提示
- 成功反馈
- 失败反馈
- 成功后跳转
- 是否有副作用

## 7.2 高危操作
删除、重置、强制下线、作废、驳回、关闭等操作必须尽量补充：

- 是否需要二次确认
- 弹窗标题
- 正文
- 确认按钮文案
- 取消按钮文案
- 是否强调不可恢复

## 7.3 列表交互
若页面为列表页，必须尽量补充：

- 默认排序字段
- 排序方向
- 分页规则
- hover 行为
- 行点击行为
- 行级操作按钮
- 是否使用“更多”收纳
- 是否支持批量选择
- 是否支持跨页选择

## 7.4 批量操作
若页面存在批量操作，必须尽量补充：

- 是否显示“当前已选 X 项”
- 是否支持全选 / 取消全选
- 是否支持跨页选择
- 是否需要所选项预览
- 是否需要二次确认
- 是否异步执行
- 是否支持仅重试失败项

## 7.5 异步任务反馈
若批量任务 / 导出 / 长耗时操作为异步执行，必须尽量补充：

- 是否展示进度条
- 是否展示预计剩余时间
- 是否展示“已成功 X 项，失败 Y 项”
- 是否提供后台任务中心入口
- 关闭当前页后是否还能查看任务结果

## 7.6 搜索与重置
若页面有搜索 / 筛选区，必须尽量补充：

- 默认筛选条件
- 点击“重置”后的逻辑
- 是清空全部，还是恢复默认态
- 重置后是否自动重新查询
- URL 参数是否同步恢复
- 分页是否回到第一页

## 7.7 副作用说明
若页面内操作会影响父页面、导航、缓存、统计卡片或关联模块，必须尽量补充：

- 会影响哪些页面 / 模块 / 区块
- 影响方式（自动刷新 / 手动刷新 / 缓存失效）
- 是否会触发异步任务
- 若副作用失败，页面如何提示

---

# 8. 异常态 / 权限态 / 空态标准

## 8.1 必须覆盖的状态
每个页面至少应尽量覆盖：

- 空态
- 加载中
- 加载失败
- 提交失败
- 删除 / 修改失败
- 无权限
- 按钮禁用
- 高危操作确认
- 数据为空但可新建的引导态

## 8.2 加载态
若页面存在加载耗时，必须尽量明确：

- 使用全局 Loading 还是骨架屏
- 骨架屏覆盖哪些区块
- 骨架屏布局是否与最终布局一致

## 8.3 权限态
必须尽量明确：

- 无权限时是整页拦截还是局部隐藏
- 仅查看角色是否仍可见操作按钮但置灰
- 是否显示无权限说明文案

---

# 9. 表格与长文案展示标准

## 9.1 表格对齐规则
默认建议：

- 金额 / 数字：右对齐
- 文本：左对齐
- 状态 / 操作：居中

## 9.2 溢出处理
必须尽量明确：

- 长文本是否打点（Ellipsis）
- 超出多少字符打点
- 是否允许换行
- Hover 是否 Tooltip 展示全量
- 头像、姓名、地址、项目名等极长内容如何处理

---

# 10. 组件复用标准

## 10.1 优先复用
Prototype Brief 必须优先推动以下区块复用统一组件：

- 搜索筛选区
- 表格 / 列表区
- 分页区
- 弹窗
- 抽屉
- 状态标签
- 空态 / 错误态
- 骨架屏
- 按钮
- 表单输入组件

## 10.2 新组件要求
若存在新组件开发，必须尽量明确：

- 新组件名称
- 解决什么问题
- 是否可沉淀为公共组件
- 是否只是一次性样式

---

# 11. 原型标注与版本标准

每份输出必须尽量补充：

- 原型版本号
- 更新时间
- 更新内容摘要
- 页面关键字段标注
- 复杂交互标注
- 条件逻辑标注
- 权限逻辑标注
- 特殊设计备注
- 浏览器 / 终端兼容要求

若为存量页面改动，建议尽量明确：

- 哪些区块为 [New]
- 哪些区块为 [Update]
- 哪些区块为 [Deprecated]

---

# 12. AI coding 工具实现提示标准

给 AI coding 工具的实现提示至少应尽量包含：

- 页面布局建议
- PC / 移动端适配范围
- 是否需要响应式
- 是否先不接真实接口
- 是否需要基础交互
- 是否优先复用现有组件
- 是否要求语义化 HTML
- 是否要求基础 A11y
- 是否要求状态变量 / 事件命名建议
- 是否建议使用多样化 mock 数据验证布局稳定性

## 12.1 A11y / 语义化 HTML
建议尽量明确：

- 使用 `<main>` `<nav>` `<section>` `<article>` 等语义化标签
- 按钮具备 `aria-label`（如适用）
- 图片具备 `alt`
- 表单控件具备 label 或等效可访问名称
- 不仅依赖颜色表达状态

---

# 13. Prototype Brief 合格检查清单

在输出 Prototype Brief 前，至少自检以下问题：

## 13.1 页面基础
- [ ] 页面名称清晰
- [ ] 页面目标清晰
- [ ] 角色边界清晰
- [ ] 页面入口 / 出口清晰
- [ ] 面包屑路径清晰

## 13.2 字段
- [ ] 关键字段完整
- [ ] 表单字段 placeholder / 示例值 / 错误提示完整
- [ ] 列表 / 详情字段一致
- [ ] 附件字段规则完整
- [ ] 敏感字段规则完整

## 13.3 交互
- [ ] 按钮状态完整
- [ ] 高危操作弹窗完整
- [ ] 搜索重置逻辑明确
- [ ] 批量操作逻辑明确
- [ ] 异步任务反馈明确
- [ ] 副作用说明明确

## 13.4 状态与异常
- [ ] 枚举值完整
- [ ] 状态颜色明确
- [ ] 空态完整
- [ ] 加载态完整
- [ ] 权限态完整
- [ ] 提交失败态完整

## 13.5 实现提示
- [ ] Design System 依赖明确
- [ ] AI coding 工具提示明确
- [ ] A11y / 语义化要求明确
- [ ] 适配范围明确
- [ ] 待确认项已列出

---

# 14. 一句话原则

**一份合格的 Prototype Brief，不只是“能看懂”，而是“能直接被 AI coding 工具、UI 和前端继续往下做”。**
