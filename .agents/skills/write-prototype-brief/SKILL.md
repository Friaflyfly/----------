# write-prototype-brief

这是一个用于**编写 Prototype Brief（原型说明稿）** 的 skill。

它的目标不是产出完整 PRD，而是把一个平台型页面或模块快速整理成一份：

- 能直接喂给 AI coding 工具（如 Cursor、Codex、Copilot 等）
- 能驱动 HTML 原型生成
- 能让产品、UI、前端快速对齐
- 字段、交互、状态、异常完整
- 可落地、可标注、可追溯

的页面级说明稿。

---

## 1. Skill Objective

你的任务是：

1. 将零散需求快速整理为页面级 Prototype Brief
2. 优先明确页面目标、角色、信息架构、模块、字段、交互、状态
3. 输出适合 AI coding 工具直接消费的结构化页面说明
4. 在信息不完整时继续推进，但必须显式标记待确认项
5. 避免一上来输出完整 PRD，保持页面级、原型驱动的颗粒度
6. 对于平台型页面，主动补足容易遗漏的字段规则、页面跳转、状态、异常和权限
7. 尽量让产物同时可用于：
   - AI coding 工具生成 HTML 原型
   - UI 继续出高保真
   - 产品与研发快速对齐

---

## 2. When to Use

在以下场景优先使用本 skill：

- 用户要快速生成 HTML 原型
- 用户希望先做页面方案，再补正式 PRD
- 用户已有会议纪要、草图、页面想法，希望先整理成页面说明稿
- 用户在做中后台 / 平台型页面设计
- 用户要给 AI coding 工具、UI、前端一个可直接消费的页面定义
- 用户要先把一个复杂需求拆成 1 个或多个页面级 brief

---

## 3. When Not to Use

以下场景不适合使用本 skill：

- 需要输出完整平台 PRD
- 需要深度描述跨模块业务规则
- 需要研发视角的逻辑审查
- 需要完整架构设计文档
- 只做视觉风格定义，不涉及业务页面结构

这些场景应改用 `write-platform-prd` 或 review 类 skill。

---

## 4. Core Principles

Prototype Brief 必须优先解决这些问题：

- 这页是谁用
- 这页要完成什么任务
- 页面由哪些区块组成
- 每个区块展示什么
- 有哪些关键字段
- 有哪些关键操作
- 操作后状态如何变化
- 空态 / 异常态 / 权限态如何表现
- 是否有角色差异
- 是否复用已有公共组件
- 是否适合直接转成 HTML 原型

不要优先写背景长文。  
不要优先写复杂业务全景。  
不要把 Prototype Brief 写成完整 PRD。  
不要只写“页面长什么样”，必须把**字段、状态、交互、异常**写清楚。

---

## 5. Output Structure

默认输出结构如下：

1. 页面名称
2. 页面目标
3. 使用角色
4. 页面入口 / 出口
5. 页面信息架构
6. 核心模块说明
7. 字段与数据展示规则
8. 关键交互与状态
9. 异常态 / 权限态 / 空态
10. 组件复用 / Design System 依赖
11. 原型标注与版本要求
12. 给 AI coding 工具的实现提示
13. 待确认项

如果需求较轻，可裁剪为 6~8 个部分。  
如果需求较复杂，可额外补充：
- 页面与上下游页面关系
- 关键枚举 / 联动规则
- 敏感字段可见性与脱敏规则

---

## 6. Writing Rules

### 6.1 页面名称
要明确页面是什么，不要写泛化名称。  
例如：
- 项目大厅页
- 订单审核页
- 收益账本详情页
- 规则配置页
- 学员入驻审核页

---

### 6.2 页面目标
必须写清楚：
- 这页解决什么问题
- 用户在这页最核心要完成什么任务
- 页面成功的标志是什么

不要只写“展示信息”“方便管理”。

---

### 6.3 使用角色
要明确：
- 谁会使用这个页面
- 不同角色是否进入同一页面
- 不同角色是否看到不同内容 / 按钮 / 字段
- 是否存在仅查看不操作的角色

若角色差异明显，必须显式写出。

---

### 6.4 页面入口 / 出口
必须尽量明确：
- 页面从哪里进入
- 页面完成操作后去哪里
- 取消按钮返回哪里
- 成功后跳转哪里
- 是否有详情页 / 编辑页 / 列表页之间的跳转关系
- 是否有弹窗、抽屉、二级页、三级页

补充要求：  
必须尽量明确当前页面在系统中的层级位置与面包屑路径。

至少尽量写清楚：
- 当前页面的面包屑路径（如：后台首页 > 交易管理 > 订单详情）
- 页面返回路径
- 页面是否可从多个入口进入
- 多入口进入时，返回逻辑是否一致

禁止出现：
- 只有入口没有出口
- 只有跳转没有返回路径
- 页面层级混乱

---

### 6.5 页面信息架构
要说明：
- 页面由哪些区块组成
- 区块顺序如何
- 哪些是主区块
- 哪些是辅助区块
- 列表页是否为“筛选区 → 列表区 → 分页区”
- 表单页是否为“标题区 → 表单区 → 操作区”
- 详情页是否分组展示

优先输出**页面骨架**，让 AI coding 工具能先搭结构。

---

### 6.6 核心模块说明
每个核心模块尽量写清楚：
- 模块名称
- 模块目标
- 模块内容
- 模块内操作
- 模块状态
- 模块之间关系

对于复杂页面，建议至少拆出：
- 搜索筛选区
- 列表 / 表格区
- 详情信息区
- 操作区
- 状态区
- 辅助提示区

补充要求：  
若页面包含搜索 / 筛选区，必须尽量明确“重置”按钮触发后的默认态逻辑。

至少尽量写清楚：
- 重置后是清空所有输入
- 还是恢复到默认筛选条件
- 默认筛选条件是什么（如默认展示近 3 个月、默认状态为全部等）
- 重置后是否自动重新查询

---

### 6.7 字段与数据展示规则
字段不要只列名字，要尽量写清楚：

- 字段名称
- 字段含义
- 展示位置
- 展示格式
- 是否必填
- 默认值
- 是否可编辑
- 是否可复制
- 是否脱敏
- 与其他字段是否联动

如果是表单字段，还需尽量明确：

- placeholder
- 正常示例值
- 错误提示文案
- 字符限制（最小 / 最大长度）
- 输入类型（文本 / 数字 / 邮箱 / 手机号 / 密码等）
- 是否允许特殊字符
- 是否允许空格

补充要求：  
若字段为附件 / 图片上传，必须尽量明确：

- 文件格式要求
- 单文件大小限制
- 总大小限制（如适用）
- 是否支持多选
- 是否支持拖拽上传
- 是否支持预览
- 是否支持下载
- 是否支持删除 / 替换
- 上传失败提示文案

补充要求：  
若页面包含表格，必须尽量定义字段对齐方式与溢出处理规则。

至少尽量明确：
- 金额 / 数字右对齐
- 文本左对齐
- 状态 / 操作居中
- 长文本默认超出隐藏（Ellipsis）
- Hover 时是否通过 Tooltip 展示全量

补充要求：  
若页面包含输入表单，必须尽量定义【即时校验】规则。

至少尽量明确：
- 是否支持失焦校验
- 哪些字段即时校验，哪些字段提交时校验
- 错误样式与错误提示出现时机
- 校验通过后错误提示如何消失

### 6.7.1 一致性要求
同一字段在以下页面必须尽量保持一致：

- 新建
- 编辑
- 列表
- 详情
- 搜索
- 筛选

禁止出现：
- 列表叫“申请人”，详情叫“提交人”
- 新建页有字段，详情页无对应展示
- 同一枚举值不同页面文案不一致

---

### 6.8 枚举值与联动规则
所有枚举字段必须尽量写清楚：

- 全部枚举值
- 对应显示文案
- 默认值
- 排序顺序
- 不同枚举值触发的页面变化

若存在联动逻辑，必须显式写出。  
例如：
- 审核状态 = 驳回 时，驳回原因必显
- 支付方式 = 微信支付 时，显示微信二维码
- 接单模式 = 报名 时，显示报名说明和审核状态

补充要求：  
若枚举值或状态需要在页面上通过视觉进行区分，必须尽量定义语义化颜色。

至少尽量明确：
- 成功态颜色
- 失败态颜色
- 警告 / 待处理态颜色
- 禁用态颜色
- 同一状态在标签、文字、图标中的颜色是否保持一致

---

### 6.9 关键交互与状态
要说明：

- 按钮做什么
- 何时可点击
- 何时置灰
- 置灰原因是什么
- 点击后发生什么
- 成功反馈是什么
- 失败反馈是什么
- 是否需要弹窗 / 抽屉 / 二次确认
- 是否需要提交中加载态
- 提交成功后跳转到哪里

若涉及列表，还需尽量说明：

- 默认排序字段
- 升序 / 降序规则
- 分页规则
- hover / 点击行为
- 行级操作按钮的位置与行为

若涉及弹窗，还需尽量说明：

- 触发条件
- 关闭方式
- 大小 / 布局
- 背景是否灰化
- 是否允许点击外部关闭

补充要求：  
涉及删除、重置、强制下线、强制关闭、作废、驳回等高危操作时，必须尽量给出具体的二次确认弹窗文案。

至少尽量明确：
- 弹窗标题
- 提示正文
- 确认按钮文案
- 取消按钮文案
- 是否需要强调不可恢复

补充要求：  
若页面涉及批量操作，必须尽量说明批量选择反馈方式。

至少尽量明确：
- 是否显示“当前已选 X 项”的计数提示
- 是否支持全选 / 取消全选
- 是否支持跨页选择
- 是否需要展示所选项预览
- 执行批量操作前是否需要再次确认影响范围

补充要求：  
若列表页行级操作按钮超过 3 个，必须尽量说明是否使用“更多”下拉菜单进行收纳。

至少尽量明确：
- 哪些操作常驻展示
- 哪些操作收纳到“更多”菜单
- 高危操作是否需要单独强调或二次确认
- 不同角色下操作列是否会变化

补充要求：  
若页面为高频操作页，必须尽量定义【快捷键】与【焦点切换】规则。

至少尽量明确：
- 是否支持 Ctrl+S / Enter / Esc 等快捷键
- 是否支持 Tab 顺序切换焦点
- 高危操作是否禁止快捷键直触发
- 快捷键是否存在角色差异

补充要求：  
若批量任务为异步执行，必须尽量明确进度反馈方式。

至少尽量写清楚：
- 是否展示执行进度条
- 是否展示预计剩余时间
- 是否展示处理中状态文案
- 是否提供跳转至“后台任务中心 / 任务列表”的入口
- 用户关闭当前页后是否还能在其他位置查看任务进度与结果

补充要求：
若页面内操作会影响父页面、列表缓存、导航结构、关联模块展示或全局配置状态，必须补充【副作用说明】。

至少尽量明确：
- 操作完成后哪些区块需要刷新
- 是否会导致父页面或列表页缓存失效
- 是否会影响侧边栏导航、Tab、统计卡片或关联模块展示
- 是否需要用户手动刷新，还是系统自动刷新
- 若刷新失败，页面如何提示

---

### 6.10 异常态 / 权限态 / 空态
必须尽量补充：

- 无数据时页面怎么表现
- 加载失败怎么表现
- 提交失败怎么表现
- 删除 / 修改失败怎么表现
- 无权限怎么表现
- 按钮禁用如何提示
- 高危操作是否有确认弹窗
- 是否存在“暂无数据，点击新建”这类引导

补充要求：  
若页面存在较长加载时间，必须尽量定义【加载形态】。

至少尽量明确：
- 使用全局 Loading 还是骨架屏（Skeleton Screen）
- 骨架屏覆盖哪些区块
- 骨架屏布局是否与最终布局一致
- 列表、详情、卡片等区域是否分别展示骨架

不要只写 happy path。

---

### 6.11 敏感字段与权限差异
若页面涉及敏感字段或高敏业务字段，必须尽量补充：

- 哪些字段需要脱敏
- 列表页、详情页、导出是否一致
- 不同角色是否看不同粒度
- 是否允许查看明文
- 查看明文是否需要二次确认或审计
- 是否允许复制、导出、截图传播

补充要求：  
若页面涉及敏感字段明文查看，必须尽量说明“查看按钮”点击后的具体交互。

至少尽量明确：
- 是直接展示，还是弹窗确认后展示
- 是否需要输入查看原因
- 是否需要二次审批或审计
- 展示后是否有时效限制
- 展示后是否允许复制、导出、截图

---

### 6.12 组件复用 / Design System 依赖
要说明：

- 是否沿用已有 Design System
- 搜索区 / 表格 / 详情区 / 弹窗 / 抽屉是否复用公共组件
- 是否涉及新组件开发
- 若是新组件，是否建议沉淀为公共组件
- 是否存在一次性页面样式
- 若偏离全局规范，原因是什么

Prototype Brief 应尽量优先推动**复用组件**，而不是制造一次性页面。

---

### 6.13 原型标注与版本要求
输出的原型说明稿应尽量补充：

- 页面关键字段说明
- 复杂交互说明
- 条件逻辑说明
- 权限逻辑说明
- 原型版本号
- 更新时间
- 更新内容摘要
- 特殊设计要求备注
- 浏览器 / 终端兼容要求（如适用）

补充要求：  
若为存量页面改动，必须尽量标识本次变更类型。

至少尽量明确：
- 哪些区域为 [New]
- 哪些区域为 [Update]
- 哪些区域为 [Deprecated]
- 是否在原型中通过角标、颜色或注释显性标识

复杂联动建议单独加“交互说明”小节。

---

### 6.14 给 AI coding 工具的实现提示
输出最后应尽量给出简洁明确的实现提示，例如：

- 页面建议使用表格布局还是卡片布局
- 是否采用响应式
- 先做静态 HTML 还是带基础交互
- 哪些模块优先静态展示
- 哪些按钮先做假交互
- 哪些状态需要 mock 数据
- 是否先不接真实接口
- 是否优先复用全局组件

补充要求：  
必须明确原型适配范围。

至少尽量明确：
- 是否仅针对 PC 端保真
- 若是 PC 端，默认保真宽度是多少（如 1440px）
- 是否需要适配移动端或平板端
- 若不做响应式，是否允许在小屏下仅保证基本可浏览
- 若做响应式，哪些模块可以折叠、换行或下沉

补充要求：  
若系统存在国际化或未来出海可能，必须尽量说明【i18n 占位】要求。

至少尽量明确：
- 文案是否允许硬编码
- 是否需要文案 Key
- Key 命名规范
- 是否预留长文案导致的排版伸缩空间

补充要求：  
若页面存在状态流转，页面状态变更应尽量与逻辑状态机一一对应。

至少尽量明确：
- 前端状态变量命名建议
- 业务枚举值与页面展示状态的映射
- 关键按钮动作与状态机事件的映射
- 是否建议按状态机拆分函数或常量

补充要求：  
生成的 HTML 必须尽量包含基础 A11y（无障碍）支持。

至少尽量明确：
- 按钮应具备可读的 aria-label（如适用）
- 图片应具备 alt
- 表单控件应具备 label 或等效可访问名称
- 交互元素应尽量使用语义化标签
- 不要仅依赖颜色表达状态

建议：
生成 HTML 原型时，建议使用多样化 mock 数据验证布局稳定性，包括：
- 极限长文案
- 空字段 / 缺失字段
- 全状态样本（覆盖所有枚举值）
- 不同长度的按钮文案
- 0 值 / 边界值 / 异常值（如适用）

目的：
- 验证表格、卡片、表单在极端情况下是否仍能正常展示
- 提前暴露省略、换行、Tooltip、空态占位等问题

这部分要面向 **AI coding 工具**（如 Cursor、Codex、Copilot 等），写得直接、具体、可执行。

---

### 6.15 待确认项
对以下内容必须显式标记：

- 角色差异未明确
- 状态流转未明确
- 枚举值未明确
- 敏感字段是否脱敏未明确
- 列表字段与详情字段关系未明确
- 页面入口 / 出口未明确
- 是否复用组件未明确
- 哪些按钮是否开放给某角色未明确

统一标记为：

- [待确认]
- [AI 推测 - 待确认]
- [实现口径待明确]

---

## 7. Default Behavior

如果用户只给了零散材料，例如：

- 一段需求描述
- 几个页面名称
- 一份会议纪要
- 一张草图
- 一段口头说明

你也应当：

1. 先识别页面类型（列表页 / 详情页 / 审核页 / 配置页 / 表单页 / 看板页等）
2. 再识别页面角色
3. 再拆页面信息架构
4. 再补字段与交互
5. 再补状态与异常
6. 再补组件复用与 AI coding 提示
7. 最后列待确认项

不要只把输入重新排版。  
不要直接输出完整 PRD。

---

## 8. Special Emphasis for Platform Pages

对于平台型页面，默认优先检查并补充：

- 页面目标是否清晰
- 页面入口 / 出口是否明确
- 面包屑与层级路径是否清晰
- 列表与详情字段是否一致
- 字段状态是否完整
- 表单 placeholder / 示例值 / 错误提示是否完整
- 附件上传字段规则是否完整
- 表格对齐与溢出规范是否明确
- 即时校验规则是否明确
- 枚举值及联动逻辑是否完整
- 状态语义化颜色是否定义
- 按钮可用 / 禁用条件是否明确
- 审核 / 配置 / 财务 / 运营角色差异
- 敏感字段脱敏
- 明文查看交互是否明确
- 空态 / 权限态 / 失败态
- 骨架屏与加载形态是否明确
- 列表排序 / 分页 / 筛选逻辑
- 搜索重置默认态是否明确
- 弹窗 / 抽屉交互
- 批量操作反馈
- 已选数量提示与所选项预览
- 行级操作列是否需要“更多”收纳
- 异步任务进度反馈是否明确
- 快捷键与焦点切换是否明确
- 页面副作用是否明确
- 页面层级跳转是否完整
- 公共组件复用
- 原型标注与版本号
- [New] / [Update] / [Deprecated] 变更标识
- 原型适配范围是否明确
- i18n 占位是否明确
- 状态机与前端状态变量映射是否明确
- A11y / 语义化 HTML 是否明确
- 适合 AI coding 工具直接生成 HTML 的模块拆分

## 9. Output Style

输出应满足：

- 结构化
- 页面级
- 可直接给 AI coding 工具使用
- 不写成冗长 PRD
- 尽量短而完整
- 尽量显性化页面逻辑
- 待确认项必须单独列出

默认输出为 Markdown。
