# write-platform-prd

这是一个用于**编写平台型 PRD** 的 skill。

它的目标是：帮助用户把复杂的平台型业务需求整理成一份**可落地、可评审、可研发、可测试、可上线演进**的 PRD。

适用于：
- 多角色协同平台
- 中后台 / 管理后台
- 审核流、接单流、履约流、结算流
- 规则配置平台
- 商家平台 / 运营平台 / 管理平台
- 强状态流转业务
- 多模块联动业务

---

## 1. Skill Objective

你的任务是：
1. 帮助用户把零散需求整理成结构化 PRD
2. 主动补足平台型 PRD 高概率缺失的部分
3. 输出研发、测试、业务都能理解的文档结构
4. 在信息不完整时，优先形成可推进评审的文档，但必须显式标注推测项与待确认项
5. 输出内容必须偏“实现导向”，而不是只讲概念或价值
6. 除功能设计外，还要兼顾技术约束、上线演进、存量兼容和系统依赖

## 2. Core Principles

平台型 PRD 必须优先关注以下维度：
- 需求背景与目标
- 角色与权限
- 核心业务对象与对象关系
- 总体流程与状态机
- 功能模块设计
- 页面 / 后台详细设计
- 字段与交互规则
- 业务规则与配置规则
- 上下游依赖与一致性
- 技术约束与非功能需求
- 上线灰度与存量兼容
- 异常场景与边界处理
- 数据口径与成功指标
- 验收标准

不要只写页面说明。不要只写主流程。不要只写前台交互。必须把“对象、角色、状态、规则、后台、异常、验收、技术边界”写进去。

## 3. Requirement Sizing｜需求分级机制

### L1：轻量迭代
适用场景：
- 文案调整
- 字段新增/删除
- 列表筛选项新增
- 小交互优化
- 小范围校验规则调整

默认输出结构：
1. 背景
2. 本次变更点
3. 影响范围
4. 交互/字段规则
5. 验收标准
6. 待确认项

### L2：标准功能
适用场景：
- 新增一个页面或一个模块内功能
- 流程有一定变化
- 涉及角色权限变化
- 涉及简单状态流转
- 需要后台支持但复杂度有限

默认输出结构：
1. 背景与目标
2. 范围
3. 角色
4. 功能设计
5. 页面/字段规则
6. 业务规则
7. 权限规则
8. 异常处理
9. 验收标准
10. 待确认项

### L3：平台型重需求
适用场景：
- 新模块
- 新流程
- 强状态机
- 多角色协同
- 规则配置
- 前后台协同
- 跨模块联动
- 涉及数据同步、灰度、存量兼容

默认输出完整结构：
1. 需求背景
2. 需求目标
3. 范围 / 非范围
4. 角色定义
5. 核心业务对象
6. 对象关系
7. 总体流程
8. 状态机
9. 功能模块设计
10. 页面 / 后台详细设计
11. 字段与交互规则
12. 业务规则与配置规则
13. 权限规则
14. 技术约束与非功能需求
15. 上下游依赖与一致性
16. 上线灰度与存量兼容
17. 异常场景与边界处理
18. 数据口径
19. 日志 / 审计 / 埋点
20. 验收标准
21. MVP 演进建议（如适用）
22. 待确认清单

### 3.1 分级裁决规则
当需求同时命中多个级别时，按**从高不从低**原则处理。

以下任一条件命中，即使用户输入很短，也应默认升级为 **L3**：
- 涉及跨模块联动
- 涉及状态机变化
- 涉及核心业务对象新增或修改
- 涉及权限模型变化
- 涉及规则配置与生效逻辑
- 涉及上下游系统依赖
- 涉及存量数据兼容、迁移或回刷
- 涉及资金、审核、履约、结算等核心链路

### 3.2 MVP 演进建议
对于 L3 需求，若预估工期较长、依赖较多、落地风险较高，建议在文末补充【MVP 演进建议】：
- 核心链路的最小闭环是什么
- 哪些能力可放入 Phase 2
- 哪些配置项可先后台兜底
- 哪些自动化能力可先人工处理
- 哪些报表 / 审计 / 批量能力可延后

## 4. Assumption Marking Rules｜AI 推测强标记规则

当用户信息不完整时，可以基于常见平台型产品逻辑补全文档，但**绝不能把推测内容混入正文当成已确认事实**。

以下内容若非用户明确提供，必须显著标记为：
- `[AI 推测 - 待确认]`
- `[待确认]`
- `[实现口径待明确]`

严禁 AI 在未获得明确输入的情况下，自行创造以下内容的具体数值或比例：
- 财务结算比例
- 分账比例
- 手续费率
- 违约赔偿金额
- 税费口径
- 法务责任边界

上述内容若缺失，必须强制标记为：
- `[财务口径待明确]`
- `[法务口径待明确]`

文档末尾必须输出：
## 【待确认清单】

## 5. Platform PRD Writing Rules

### 5.1 需求背景与目标
必须写清楚：为什么做、当前痛点、本次解决什么、本次不解决什么、目标是提效/控风险/支撑新业务等。

### 5.2 角色与权限
尽量拆清楚：
- 页面权限
- 按钮权限
- 字段权限
- 数据权限
- 导出权限
- 配置权限
- 强制操作权限

### 5.3 核心业务对象与关系
必须尽量识别业务对象，如项目、订单、任务、审核单、结算单、账户、配置项、组织、规则等，并明确：
- 对象定义
- 生命周期
- 与其他对象的关系
- 是否有跨模块同步

凡是存在父子对象、主从对象、依附关系或引用关系，必须定义【生命周期联动】规则，避免孤儿数据。

### 5.4 总体流程与状态机
凡是涉及创建、提交、审核、驳回、生效、失效、执行中、完成、关闭、取消的流程，都必须补状态机。

必须写清楚：
- 状态枚举
- 状态含义
- 状态迁移条件
- 各状态允许操作
- 状态迁移后的系统动作
- 是否可逆
- 是否影响其他模块状态

若涉及审核、履约、结算、记账、资源发放、配置发布等流程，必须显式说明【逆向流转定义】：
- 哪些状态允许撤回 / 撤销
- 哪些状态一旦生效后不可直接撤回
- 是否需冲抵 / 冲正 / reversal 单据
- 冲抵单 / 冲正单的生成规则和状态变化

### 5.5 页面 / 后台详细设计
必须兼顾前台与后台：
- 页面目标、结构、关键操作、空态 / 失败态 / 权限态
- 后台配置项、查询条件、批量操作、强制操作、导出能力、日志查看

凡是涉及中大数据量列表（尤其是百万级及以上），必须定义【查询性能约束】：
- 默认查询条件
- 强制过滤项
- 是否允许无条件全量查询
- 精确搜索 / 模糊搜索字段
- 无索引字段模糊搜索限制
- 导出限制

同时应尽量说明【组件化规范】与【Design System 依赖】。

### 5.6 字段与交互规则
需要尽量写清楚：
- 字段名称、含义、输入类型、是否必填、默认值、是否可编辑、枚举值、校验规则、联动关系、展示规则
- 按钮显示条件、置灰条件、点击动作、成功反馈、失败反馈、是否二次确认

若字段涉及 PII 或高敏业务信息，必须定义【字段级可见性与脱敏规则】：
- 列表页、详情页、导出、接口返回是否脱敏
- 脱敏规则
- 不同角色的脱敏粒度是否不同
- 是否允许查看明文
- 查看明文是否需要二次确认 / 审批 / 审计

### 5.7 业务规则与配置规则
至少尽量写清楚：
- 触发条件
- 判断条件
- 优先级
- 限制条件
- 生效范围
- 是否可配置
- 配置后何时生效
- 是否影响历史数据
- 冲突规则谁覆盖谁

若涉及规则配置、策略配置、阈值配置、审核配置等，必须定义【生效验证机制】：
- 配置保存后是否需要发布才生效
- 是否区分草稿版本与线上生效版本
- 是否支持查看版本差异
- 是否支持配置前预校验
- 配置发布后如何验证已生效
- 缓存刷新或配置同步时效

### 5.8 上下游依赖与一致性
必须尽量写清楚：
- 上游 / 下游系统
- 依赖接口 / 消息 / 定时任务
- 主数据源 / 从数据源
- 强一致 / 最终一致 / 异步补偿
- 同步失败是否阻塞主流程
- 重试、补偿、人工兜底

若涉及接口升级、字段变更、协议调整或新老逻辑切换，必须定义【版本兼容性】与【灰度控制】：
- 新老接口并存策略
- 平滑过渡策略
- 新老字段并存与废弃策略
- 下游如何识别新老版本
- 下游未升级时如何兼容

### 5.9 技术约束与非功能需求
平台型 PRD 尽量补充：
- 预估数据量级
- 峰值并发 / QPS
- 页面 / 查询 / 批量 / 导出时效要求
- 数据生命周期（TTL）
- 日志和审计保留周期
- 归档 / 冷存 / 删除策略
- 合规要求
- 稳定性与可用性要求

涉及跨地域 / 跨时区业务时，必须定义时间口径：
- 存储时区
- 展示时区
- 统计时区
- 结算时区
- 定时任务触发时区
- 导出口径时区

### 5.10 异常场景与边界处理
必须主动补充：
- 空数据
- 加载失败
- 提交失败
- 接口超时
- 权限不足
- 重复提交
- 状态冲突
- 并发修改
- 历史数据兼容
- 跨模块同步失败
- 配置缺失
- 数据被删除 / 失效

凡是可能被多人同时操作的场景，必须显式说明【并发冲突策略】：
- Last Writer Wins / 乐观锁 / 悲观锁 / 自动排队
- 冲突提示文案
- 冲突日志
- 是否允许后台强制覆盖

凡是存在撤回、撤销、冲抵、冲正等逆向动作，必须补充失败场景与边界处理。

### 5.11 数据口径与成功指标
必须尽量写清楚：
- 指标名称
- 分子
- 分母
- 统计范围
- 统计时点
- 统计延迟
- 更新频率
- 是否去重
- 页面口径与导出口径是否一致

凡是平台型需求涉及流程优化、效率提升、自动化替代、准确率提升、风险控制或业务转化，必须补充【成功指标（Success Metrics）】。

### 5.12 日志 / 审计 / 埋点
建议默认考虑：
- 操作日志
- 审计日志
- 高危操作记录
- 状态流转日志
- 导出日志
- 配置变更日志
- 关键业务埋点

若涉及敏感信息明文查看、导出、复制、解密展示，必须定义敏感信息访问审计要求。

### 5.13 验收标准
验收标准不能只写“功能正常”。必须尽量写成可验证内容：
- 给定什么前置条件
- 执行什么操作
- 页面应显示什么
- 数据应如何变化
- 状态应如何变化
- 权限应如何生效
- 异常情况下如何提示
- 后台是否有记录
- 跨模块是否同步成功

若涉及核心链路、大批量数据、跨模块联动、灰度上线或存量兼容，还应尽量补充非功能验收点。

## 6. Default Behavior

如果用户只给了零散材料，例如会议纪要、原型说明、页面清单、功能点列表、业务规则草稿或口头描述，你也应当：
- 先判断需求级别（L1/L2/L3）
- 按“从高不从低”原则裁决复杂度
- 先抽取业务目标
- 再识别角色
- 再识别业务对象
- 再抽总体流程
- 再补状态机
- 再写功能模块
- 再补字段与规则
- 再补后台与异常
- 再补技术约束、依赖、兼容
- 最终整理成平台型 PRD

当需求中出现手机号、身份证号、银行卡号、姓名、地址、邮箱、结算金额、税号等字段时，默认触发 PII 风险检查，并主动追问或标记待确认项。

## 7. Special Emphasis for Platform Products

对于平台型需求，默认优先检查并补充以下内容：
- 多角色与多端协同
- 对象模型
- 状态机闭环
- 逆向流转与后悔药逻辑
- 规则可配置性
- 配置生效验证与线上版本治理
- 数据口径统一
- 多时区与国际化时间口径
- 成功指标可度量性
- 对象生命周期联动与防孤儿数据
- 权限分层与权限变更可审计
- PII / 敏感字段分级可见与脱敏策略
- 明文查看的二次确认与审计要求
- 并发冲突策略
- 批量任务闭环与断点恢复
- 长列表查询性能约束
- 组件化规范与 Design System 依赖
- 状态变化与任务结果触达
- 数据生命周期
- API First 能力暴露
- 接口版本兼容与平滑升级
- 上线使能与操作手册同步
- 财务 / 法务高压线禁脑补
