# write-prototype-brief 使用说明

这个文件不是正式 PRD 模板。  
它的作用是：帮助大家把一个页面级需求，快速整理成适合 AI coding 工具直接消费的 **Prototype Brief（原型说明稿）**。

适用对象：
- 产品经理
- UI / 交互设计师
- 前端
- 使用 Cursor、Claude、Copilot 等工具做页面原型的人

---

## 这个 skill 是干什么的

`write-prototype-brief` 用于把一个页面或一组页面的需求，整理成一份：

- 页面目标清晰
- 区块结构明确
- 字段规则完整
- 状态与交互可落地
- 异常态 / 权限态已覆盖
- 可直接喂给 AI coding 工具

的页面说明稿。

它不是完整 PRD，也不是单纯写页面文案。  
它的定位是：

**业务需求 ↔ HTML 原型**

之间的中间层。

---

## 它最适合解决什么问题

很多时候不是“不会画页面”，而是：

- 页面目标没讲清
- 谁用、谁能点、谁能看不清楚
- 状态变化没写
- 列表字段和详情字段对不上
- 批量操作、异步任务、搜索重置逻辑没写
- AI coding 工具只能猜，最后生成的 HTML 返工很多

这个 skill 的目标，就是先把这些页面级问题收口。

---

## 和 write-platform-prd 的区别

### `write-prototype-brief`
适合前置使用，先把页面方案收出来。  
重点解决：
- 页面目标
- 页面骨架
- 字段
- 交互
- 状态
- 异常
- 实现提示

### `write-platform-prd`
适合后置使用，在页面方案较清楚后再补正式文档。  
重点解决：
- 业务对象
- 规则
- 状态机
- 权限模型
- 上下游依赖
- 灰度兼容
- 数据口径
- 验收标准

一句话理解：

**先用 Prototype Brief 把页面做出来，再用 Platform PRD 把系统逻辑补完整。**

---

## 适合什么场景

这个 skill 特别适合以下类型的页面：

- 列表页
- 详情页
- 新建 / 编辑页
- 审核页
- 配置页
- 工作台 / 看板页
- 协同页
- 账本 / 明细页
- 批量任务页

尤其适合：
- 中后台
- 管理后台
- 运营后台
- 商家后台
- 多角色页面
- 强流程页面

---

## 不适合什么场景

以下内容不建议用这个 skill：

- 完整平台型 PRD
- 研发逻辑审查
- 纯技术架构设计
- 纯视觉规范
- 单纯写品牌文案
- 纯 C 端活动页创意稿

---

## 可以直接拿什么材料来写

不需要固定模板。  
以下材料都可以直接拿来喂：

- 一段页面需求描述
- 会议纪要
- 飞书文档片段
- 草图
- 页面清单
- 原型想法
- 已有页面截图 + 改版诉求
- PRD 里的一小段页面说明

也就是说：

**材料不一定完整，但至少应该能判断页面目标。**

---

## 输入不完整时会怎么处理

这个 skill 在信息不完整时不会停住，而会这样推进：

1. 先识别页面类型
2. 先抽页面目标
3. 再识别角色
4. 再拆页面信息架构
5. 再补字段、交互、状态、异常
6. 最后把不确定的部分列入待确认项

待确认内容会统一标注为：

- `[待确认]`
- `[AI 推测 - 待确认]`
- `[实现口径待明确]`

所以你不需要等所有信息齐全了再开始写。

## 怎么理解页面复杂度

### L1：轻页面
适合：
- 简单列表页
- 小表单
- 文案调整
- 筛选项新增
- 小交互优化

输出更短，重点是：
- 页面区块
- 字段
- 按钮
- 异常态

### L2：标准页面
适合：
- 新建 / 编辑页
- 一般详情页
- 基础审核页
- 标准配置页

会重点写：
- 角色
- 字段
- 交互
- 状态
- 异常
- 实现提示

### L3：复杂平台页面
适合：
- 多角色页面
- 强状态页面
- 配置发布页
- 批量任务页
- 财务 / 审核 / 权限类页面
- 有敏感字段或异步任务的页面

会重点补：
- 面包屑与入口出口
- 字段一致性
- 语义化颜色
- 批量操作反馈
- 异步任务进度
- 搜索重置默认态
- 明文查看交互
- 副作用说明
- A11y / 语义化 HTML

## 推荐怎么喂给 skill

### 方式一：直接扔原始材料
最常见、也最实用。  
直接丢会议纪要、页面描述、原型想法，让它整理出一版 Prototype Brief。

### 方式二：补一句重点关注方向
比如：

- 这页重点看状态和按钮关系
- 这页重点看审核退回逻辑
- 这页重点看批量任务反馈
- 这页重点看 PII 明文查看
- 这页重点看搜索筛选和重置逻辑

这样输出会更聚焦。

### 方式三：指定页面类型
比如明确说：
- 这是一个列表页
- 这是一个配置页
- 这是一个审核页
- 这是一个详情 + 抽屉编辑页

这样更容易输出稳定结构。

---

## 哪些信息补充后会明显更准

虽然不是必须，但如果你有这些信息，输出会更准：

- 页面名称
- 页面目标
- 使用角色
- 页面入口
- 页面出口
- 是否存在状态流转
- 是否有敏感字段
- 是否有批量操作
- 是否有异步任务
- 是否有搜索筛选区
- 是否有默认筛选口径
- 是否有 Design System
- 是否只做 PC 端原型

---

## 用这个 skill 时特别要留意什么

### 1. 它不是完整 PRD
不要指望它一次性把跨模块系统逻辑都讲完。  
它的重点是页面级，而不是系统级。

### 2. 角色差异一定要看
很多页面问题不是布局问题，而是：
- 谁看到哪个按钮
- 谁能改
- 谁只能看
- 谁能查看明文

### 3. 状态和按钮关系一定要看
不要只看页面长什么样，要看：
- 什么状态下按钮能点
- 点完后会去哪
- 成功 / 失败怎么反馈
- 是否会影响其他页面

### 4. 待确认项一定要收口
如果页面里有：
- `[待确认]`
- `[AI 推测 - 待确认]`
- `[实现口径待明确]`

这些不要直接丢给研发或 AI coding 工具当事实使用。

---

## 输出结果怎么用

这个 skill 的输出通常可以这样使用：

1. 直接喂给 Cursor / Claude / Copilot 生成 HTML 原型
2. 交给 UI 补高保真
3. 交给前端先搭静态页
4. 用于页面评审
5. 收敛后，再补正式 `write-platform-prd`

---

## 最佳实践

### 最稳的用法
1. 先丢原始材料生成 Prototype Brief
2. 重点检查待确认项
3. 再喂给 AI coding 工具出 HTML
4. 看原型评审一轮
5. 需要的话再补正式 PRD

### 最容易翻车的用法
- 一句“帮我做个后台页面”直接让 AI coding 工具生成
- 不写角色
- 不写状态
- 不写异常态
- 不写默认态 / 重置逻辑
- 不看待确认项

---

## 一句话原则

**这个 skill 不是帮你“把页面说得更漂亮”，而是帮你把平台型页面最容易遗漏的字段、状态、交互、异常和实现提示先显性化。**
